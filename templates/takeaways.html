50 takeaways across 5 categories 4,308 conversations distilled

Two years. 4,308 conversations. 130,000 pages. Ten data sources spanning ChatGPT, Claude, Gmail, SMS, Facebook, Twitter, and half a dozen other formats. The database that holds all of this — Megabase — is itself a product of the work it catalogues.

These 50 takeaways are not tips. They are hard-won lessons from building a personal knowledge archaeology system with large language models as collaborators. Some were obvious in hindsight. Others were counterintuitive. A few were painful.

The Five Categories

Prompting (1-15): How to communicate with a model so it produces what you actually need. The gap between "ask a question" and "design an interaction" is where most of the value lives.

Workflow (16-25): How to organize your time, your output, and your expectations around AI collaboration. The model is a tool; you are the craftsperson.

Architecture (26-35): How to build systems that use LLMs without depending on them. SQLite as interchange format. Chunking as fundamental operation. The sidecar pattern.

Mindset (36-45): The psychological and philosophical shifts that come from sustained AI collaboration. You are the curator. The process is the product. The database is autobiographical.

Failure Modes (46-50): The ways it goes wrong. Scope creep, premature abstraction, hallucinated confidence, ID fragility, and the rabbit hole that has no bottom.

How to Read This

Each takeaway is a card. The number is its rank within its category, not its importance. Every takeaway links to the conversations where the insight crystallized. Browse linearly or filter by category — the tab above gives you both options.

1 Prompting

Start with structure, not content.

Give the LLM a format before you give it a task. Headers, bullet points, tables — the shape of the output is more important than the instructions about what to put in it.

2 Prompting

The first response is a draft, not an answer.

Always iterate. The model's first attempt is a starting point for refinement, not a finished product. Plan for 2-3 rounds minimum.

3 Prompting

Chain of thought works because it forces sequencing.

The model thinks better when you make it show its work. "Think step by step" isn't magic — it prevents the model from skipping logical dependencies.

4 Prompting

Personas are cognitive primes, not roleplay.

"You are a senior engineer" changes reasoning quality because it activates a different distribution of training data. It's not pretend — it's retrieval.

5 Prompting

Short prompts get short answers.

Length of input correlates with depth of output. If you want a thorough response, provide thorough context.

6 Prompting

"Give me 20 ideas" beats "give me an idea" 20 times.

Batch requests produce more varied output because the model avoids repeating itself within a single response.

7 Prompting

Constraints produce creativity.

"Write a poem in exactly 14 lines" gets better results than "write a poem." Constraints focus the model's generation and reduce generic output.

8 Prompting

The prompt is a UI.

Treat it like interface design: what does the model need to see, in what order, with what emphasis? Prompt engineering is information architecture.

9 Prompting

Examples are worth a thousand words of instruction.

One good example beats three paragraphs of description. Few-shot prompting works because the model pattern-matches on structure.

10 Prompting

Temperature is a collaboration dial.

Low for precision tasks, high for brainstorming, medium for drafting. Most people never touch it. They should.

11 Prompting

System prompts are persistent context.

Use them for persona and constraints, not step-by-step instructions. They set the frame; user messages set the task.

12 Prompting

The model can't see what you didn't paste.

If it's not in the context window, it doesn't exist for the model. Don't assume shared knowledge — provide it.

13 Prompting

Negations confuse.

"Don't use jargon" works worse than "Use plain language." Tell the model what TO do, not what to avoid.

14 Prompting

Markdown structure improves reasoning.

Headers and lists in prompts produce structured output. The model mirrors the organization you provide.

15 Prompting

The best prompt is often a conversation.

Multi-turn refinement beats a single perfect prompt. Use the first turn to orient, the second to refine, the third to polish.

16 Workflow

Use the LLM for the 80%, then do the 20% yourself.

The last mile of quality is human. The model gets you to draft-quality fast; you provide the judgment.

17 Workflow

Batch similar tasks.

Don't prompt one conversation at a time — export 20 and process them together. Batching amortizes context-setting cost.

18 Workflow

Version your prompts like code.

Keep the working version, archive the old ones. Prompts drift and degrade. Version control catches regressions.

19 Workflow

LLMs are force multipliers, not replacements.

They amplify expertise — they don't substitute for it. A domain expert with an LLM beats an LLM alone every time.

20 Workflow

Context switching kills LLM productivity too.

Keep focused sessions on one topic. A 40-message deep dive beats 40 one-message questions across different topics.

21 Workflow

Export early, export often.

Don't let good output live only in chat history. Export to files, databases, documents. Chat history is ephemeral; artifacts persist.

22 Workflow

The subscription is the budget.

Design workflows that use existing subscriptions (ChatGPT Plus, Claude Pro), not per-token API calls. Zero marginal cost changes behavior.

23 Workflow

Build tools to talk to the LLM, not tools that ARE the LLM.

The interface layer matters. Dreambase's chunker prepares data FOR ChatGPT, not inside ChatGPT.

24 Workflow

Read the output critically.

Fluent text does not equal correct text. LLMs are confident about everything. The more polished the prose, the harder you should check the facts.

25 Workflow

Keep a research journal.

Your conversations are data about your own thinking. Dreambase exists because the conversation history turned out to be the most valuable artifact.

26 Architecture

Chunking is the fundamental operation.

If you can chunk it, you can process it. Every large-scale LLM workflow starts with splitting content into context-window-sized pieces.

27 Architecture

SQLite is the universal interchange format.

Put everything in a database first, ask questions later. Ten sources, six formats, one megabase.db. SQL is the query language for life.

28 Architecture

Deterministic first, probabilistic second.

Keyword matching before LLM classification. VADER sentiment before GPT-4 analysis. Cover 80% for free, then spend tokens on the hard 20%.

29 Architecture

The sidecar pattern works.

Keep LLM-generated content in separate files from code. Narratives in JSON, templates in HTML. Either can change without breaking the other.

30 Architecture

Curated IDs are the simplest recommendation engine.

A human-curated list of 20 conversation IDs beats a vector similarity search. Curation is taste; similarity is statistics.

31 Architecture

FTS5 is a superpower.

Full-text search on 4M messages in milliseconds, no external service needed. SQLite's FTS5 is the most underrated tool in the stack.

32 Architecture

Flask + SQLite + Jinja2 is a complete stack.

You don't need React for a personal knowledge tool. Server-rendered HTML with a good CSS file does 95% of what an SPA does with 5% of the complexity.

33 Architecture

Hardcoded data is fine for small scale.

30 scholars in a Python dict is simpler than a CMS. Don't build infrastructure for scale you'll never reach.

34 Architecture

The rabbit hole is the UI pattern.

Every page should link to related pages. That's the whole UX. Navigation is discovery. Links are the product.

35 Architecture

Build the database before the interface.

Prove the data works in SQL before touching HTML. If the queries don't return interesting results, no amount of CSS will save you.

36 Mindset

You are the curator, the model is the librarian.

You decide what matters; it finds and organizes. The creative judgment is yours. The scale is the model's.

37 Mindset

Intellectual archaeology is a real discipline.

Your old conversations are primary sources. Treating them as data — tagging, indexing, searching — reveals patterns invisible in real-time.

38 Mindset

Cross-domain connections are the highest-value output.

Alchemy into game design is more interesting than either alone. The LLM excels at bridging domains because it has no disciplinary loyalty.

39 Mindset

The process IS the product.

Building Dreambase taught more than reading any book about knowledge management. The act of organizing is the act of understanding.

40 Mindset

Obsessive documentation pays compound interest.

Today's notes are tomorrow's database entries. The conversations you export now become the showcase pages of next month.

41 Mindset

Personal tools don't need to be products.

Build for yourself first. Generalize only if someone else asks. The best personal tools are opinionated and idiosyncratic.

42 Mindset

The conversation is the unit of thought.

Not the message, not the session — the full conversation arc. A 400-page conversation has a narrative structure that individual messages don't.

43 Mindset

Sentiment analysis reveals what you can't self-report.

Your excitement about alchemy is measurable. VADER scores on user messages show which topics produce the most positive engagement — data about passion.

44 Mindset

Tags are a lossy compression of meaning.

But they're the best compression we have. 9 tags across 3,309 links create a navigable map of 4,308 conversations. Imperfect but indispensable.

45 Mindset

Two years of LLM conversations is a portrait.

The database is autobiographical. Topics, frequencies, sentiment, depth — they paint a picture of who you are and what you care about.

46 Failure Modes

Scope creep is the default.

You will always want one more feature. Use a parking lot document. Write the idea down, don't build it now.

47 Failure Modes

Premature abstraction wastes time.

Build three concrete things before abstracting. A reusable component that's only used once is dead weight.

48 Failure Modes

Re-ingestion destroys IDs.

If you rebuild the database, every hardcoded conversation link breaks. This is the single most dangerous failure mode in Dreambase's architecture.

49 Failure Modes

LLMs hallucinate confidence.

The more certain the output sounds, the more you should verify. Fluency is not accuracy. Polish is not truth.

50 Failure Modes

The rabbit hole has no bottom.

Set a timer. Ship something. Come back tomorrow. The intellectual journey never ends, but the engineering sprint must.

How LLM usage evolved over 2 years

Early 2024
Simple Q&A
Single-turn questions. "What is X?" "How do I Y?" The model as encyclopedia. Output consumed and discarded — no archiving, no system.
Mid 2024
Extended Conversations
Multi-turn deep dives. 50-100 message conversations on alchemy, philosophy, game design. The model as research partner. First signs of the database impulse — "I should save these."
Late 2024
Project Building
Using LLMs to build actual software: MTG overlays, tarot engines, database tools. The model as pair programmer. Prompt engineering becomes a real skill — learning to chunk, iterate, validate.
Early 2025
Knowledge Systems
Building systems about the conversations themselves. Megabase conceived. The model helping to organize its own output. Meta-cognitive recursion: using AI to understand how you use AI.
Mid 2025
Batch Processing Workflows
Chunker produces batch files. Summarizer runs in parallel. The model integrated into data pipelines, not just conversations. Deterministic + probabilistic hybrid approach crystallizes.
Late 2025 — 2026
Agentic Swarms
Claude Code with 33 PKD skills. Multi-agent orchestration. The model as a team of specialists, each with defined roles and communication protocols. The Eldritch Swarm architecture.
