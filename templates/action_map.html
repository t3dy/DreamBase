Fix the 3 Sidecar JSON Bugs

2 min · 4 lines of code · prevents 500 errors
The sidecar JSON loader in app.py has no try/except. A malformed JSON file crashes the entire scholar or showcase page.
What's the bug?

In app.py lines 1072-1083 (showcase route) and 1244-1255 (scholar route), the code does json.load(f) without exception handling. If anyone saves a sidecar file with a trailing comma or unclosed bracket, the entire route returns HTTP 500.

The fix

Wrap the json.load() in both locations with try/except json.JSONDecodeError and fall back to an empty dict. Four lines of code, two locations.

Why it matters

As you start creating sidecar JSON files (narratives, timelines, quotes for the 30 scholars and 3 showcases), the probability of a malformed file approaches 1. This fix is the cheapest possible reliability improvement.

Build the Reader Pane

2 hours · new route + AJAX panel · high user impact
Click "1,438 pages" on any scholar page and read all linked conversations inline. The most-requested feature.
Design decision: three options

Option A: Side panel (Gmail-style). A 400px right panel slides in when you click the page stat. The main page stays visible. Good for reference browsing. Con: narrow reading width on small screens.

Option B: Full-page takeover. A new route /read/scholar/pico-della-mirandola that shows all messages in a scrollable book-like view. Good for deep reading. Con: loses the scholar page context.

Option C: Expandable accordion. Below the stats bar, an expandable section shows a conversation picker, then messages. Stays on the same page. Good for quick reference. Con: pushes other content down.

Recommended: Option A with lazy loading

Build a /api/read/<conv_id> JSON endpoint that returns paginated messages. The scholar page gets a slide-in panel that loads conversations one at a time. User picks which conversation to read from a list sorted by page count. Messages load in batches of 50 (already the pattern in conversation.html).

Technical details

The main risk is performance. Pico has 1,438 pages across 6 conversations. Loading all messages at once would be ~100,000 DOM nodes. Solution: load one conversation at a time, paginate messages, lazy-load on scroll. The existing messages table has an index on conversation_id so queries will be fast.

Alchemy Auto-Hyperlinking

3 hours · JS + templates · site-wide rabbit holes
After AlchemyDB is ingested, make every alchemy term on every page clickable. A tooltip shows the definition; clicking goes to the full lexicon entry.
The technical challenge

Auto-hyperlinking means scanning rendered HTML text for 187 terms and wrapping matches in <a> tags. This must happen client-side (after template rendering) or server-side (in Jinja2 filters).

Option A: Client-side JavaScript

Load the term list as a JSON array. On page load, walk the DOM text nodes and replace matches. Pros: works on every page with one shared script. Cons: flash of unstyled content, performance on long pages.

Option B: Jinja2 custom filter

Create a |alchemize filter that scans text and wraps terms. Apply it to summary fields, pedagogy boxes, and narrative text. Pros: no FOUC, server-side rendering. Cons: must be applied explicitly in each template.

Recommendation

Option B (Jinja2 filter) for structured content (summaries, pedagogy). Option A (JavaScript) as a fallback for dynamic content. The Jinja2 filter is more reliable and doesn't cause layout shifts.

Blast radius warning

This touches ALL templates. Per the Failure Audit, it should run LAST — after all other template changes are stable. Test on one page (e.g., Paracelsus scholar page) before rolling out.

Architecture & Scale

Move Curations to Database

half day · schema change · solves the ID fragility problem
The Failure Audit's #1 risk: if the database is rebuilt, all hardcoded conversation IDs break. Move SCHOLARS, COLLECTIONS, SHOWCASES into a curations table.
The problem

200+ conversation IDs are hardcoded in Python dicts in app.py. If anyone re-runs the ingestors, SQLite auto-increments from scratch, and every curated page shows empty. This is the single most fragile part of the architecture.

The solution

Create a curations table: id, type (scholar/collection/showcase), slug, title, data_json. Store the conversation_ids, tags, pedagogy, etc. in a JSON blob. The Python dicts become seed data that's loaded into the DB once. Routes query the curations table instead of reading from memory.

Migration path

Write a migrate_curations.py script that reads the current dicts from app.py and inserts them into the new table. Update routes to query the DB. Keep the dicts as fallback during transition.

Why not now?

This is a refactor that doesn't add features. It's important for long-term reliability but not urgent while the database isn't being rebuilt. Do it before any re-ingestion.

GitHub Pages Static Export

half day · freeze script · makes Dreambase publicly accessible
Export the Flask app as static HTML for GitHub Pages hosting. Every page becomes a .html file. Search becomes client-side.
What changes

Flask-Freeze or a custom script renders every route to a .html file. FTS search becomes a client-side lunr.js index (pre-built from conversation titles + summaries). Database queries become pre-computed JSON files loaded by JavaScript.

What stays the same

All templates, CSS, card layouts, tab switching. The static site looks identical to the Flask app for read-only browsing. Only search and the conversation detail page (message pagination) need client-side alternatives.

Trade-offs

Pro: free hosting, shareable URL, no server needed. Con: search quality degrades (lunr.js vs FTS5), no message-level browsing (too many pages to pre-render), larger initial download (pre-computed JSON).

Vector Embeddings + Semantic Search

1 day · ChromaDB · "find conversations like this one"
Add vector embeddings to enable semantic search: "find conversations about transformation" returns alchemy, game design, and personal growth conversations.
How it complements FTS5

FTS5 finds exact words. Vector search finds meaning. "Transformation" in FTS5 returns conversations containing that word. With embeddings, it also finds conversations about transmutation, metamorphosis, character development, and code refactoring — all "transformation" in different domains.

Implementation

Use ChromaDB (already used in the SMS project). Embed conversation summaries (4,243 texts × ~100 tokens each = 424K tokens). At OpenAI embedding prices ($0.02/1M tokens): about $0.01 total. Store embeddings in a ChromaDB collection alongside megabase.db.

UI integration

Add a "similar conversations" section to every conversation detail page. Add a "semantic search" toggle to the browse page search bar. When enabled, search queries are embedded and compared against the collection.
